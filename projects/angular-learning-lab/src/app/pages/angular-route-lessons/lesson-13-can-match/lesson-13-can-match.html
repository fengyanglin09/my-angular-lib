canMatch Guards

canMatch decides whether a route branch is eligible to match. This is especially useful in front of lazy feature routes because Angular can read the route's access metadata and redirect before activating the protected branch.

When To Use canMatch

Main question
Should this route branch be considered a match?
Use for
Feature flags, protected lazy feature entrances, role-based route branches.
Lazy route reason
One canMatch guard can protect the entrance to a whole lazy feature group.
Use canActivate inside
When the feature is available, but one child page needs an extra permission.
Not for
Unsaved changes when leaving; use canDeactivate for that.

canMatch Sequence

  1. Navigation starts.
  2. Angular sees a possible protected route branch.
  3. canMatch checks whether that branch may match.
  4. If allowed, Angular matches the branch and can load its lazy children.
  5. If denied, Angular skips or redirects before entering that branch.

What To Remember

  1. canMatch runs while Angular is matching a route branch.
  2. Route data is a good place to declare what role or permission the branch requires.
  3. The guard should read that metadata and use a service to check the current user.
  4. Return true to let that branch match.
  5. Return a UrlTree to redirect somewhere else.
  6. canMatch is useful before lazy feature routes.
  7. At a lazy parent route, canMatch protects the entrance to the whole feature route group.
  8. Inside that lazy group, canActivate can still protect one specific child page.
  9. canActivate protects activation; canMatch protects matching.
  10. Use canMatch when the whole route branch should feel unavailable unless access rules pass.
  11. Use canActivate when the route is valid, but entering that matched page needs a permission check.
  12. Use canDeactivate for the opposite direction: leaving a page with unsaved or unfinished work.
  13. Changing access state later does not automatically remove a user from an already active route.
