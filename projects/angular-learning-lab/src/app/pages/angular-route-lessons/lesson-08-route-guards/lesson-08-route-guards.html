Route Guards

Route guards decide whether navigation should continue before Angular activates a route. In real applications, the route often declares what access it requires with data, and the guard contains the reusable checking logic.

Guard Return Values

Route data
requiredRole: 'Admin'
true
Allow navigation and activate the target route.
false
Cancel navigation and stay where you are.
UrlTree
Cancel the attempted route and redirect to the UrlTree route.

When To Use canActivate

Main question
May this user enter this matched page?
Use for
Login checks, role checks, onboarding gates, page-level permissions.
Not for
Unsaved changes when leaving; use canDeactivate for that.

canActivate Sequence

  1. Navigation starts.
  2. Angular matches the target route.
  3. canActivate reads route data and checks access.
  4. If allowed, Angular activates the route and renders its component.
  5. If denied, Angular cancels or redirects before rendering the target component.

What To Remember

  1. canActivate runs before Angular activates the guarded route.
  2. A functional guard can use inject(...) to read services.
  3. Route data is a good place to declare access requirements such as requiredRole.
  4. The guard should contain reusable checking logic, not one hardcoded rule per route.
  5. Return true to allow navigation.
  6. Return false to cancel navigation and stay on the current route.
  7. Return a UrlTree when you want to redirect instead of merely canceling.
  8. A guard redirect can be absolute, or relative when you pass relativeTo: route.
  9. Relative guard redirects depend on the route snapshot passed into the guard.
  10. Use canActivate when the page route is valid, but entering it requires access.
  11. Use canMatch when the route branch itself should not match unless access is valid.
  12. Use canDeactivate when the user is leaving a route and might lose work.
  13. Guards run during navigation; they do not continuously watch access state after activation.
