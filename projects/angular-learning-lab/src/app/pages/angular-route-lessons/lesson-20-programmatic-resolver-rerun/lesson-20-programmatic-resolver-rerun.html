Programmatically Rerun Resolvers

Resolvers do not have a direct rerun method. To rerun one from code, perform a navigation that changes a URL value the route is configured to watch.

Pattern Summary

Button click
Calls router.navigate([]) relative to the current route.
URL change
Only refreshId changes; the same component can stay on screen.
Resolver rerun
paramsOrQueryParamsChange tells Angular this query change matters.

Resolver Rerun vs Service Reload

Use resolver rerun
When route-owned data should be represented by navigation and route data.
Use service reload
When the page is already active and you only need to refresh an internal panel.
Rule of thumb
If the URL should explain the reload, navigation is a good fit.

Mental Model

click refresh
  navigate on same route
  change refreshId query param
  router sees query param change
  route policy allows rerun
  resolver runs again

What To Remember

  1. There is no direct resolver.run() API.
  2. Resolvers rerun through router navigation.
  3. A refresh query param is a common way to intentionally trigger same-page navigation.
  4. runGuardsAndResolvers: 'paramsOrQueryParamsChange' is needed when a query param should rerun the resolver.
  5. Use this pattern when the refreshed data belongs to the route; use a service reload when the refresh is only local UI state.
