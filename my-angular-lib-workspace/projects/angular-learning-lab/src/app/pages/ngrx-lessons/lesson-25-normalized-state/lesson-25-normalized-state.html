Normalized State And Selector Joins

Real applications often receive related data from a backend. Instead of nesting duplicate author objects inside every book, store authors and books separately, then join them in selectors.

Why This Pattern Matters

  1. Each author is stored once, even if they have many books.
  2. Each book stores only authorId, like a database foreign key.
  3. Selectors join books with author names for the UI.
  4. Filtering by selected author changes only one small state value.
  5. The component receives simple display-ready book view models.
